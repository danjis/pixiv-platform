## 一、开场（1 分钟）

各位老师好，我是 XXX，我的毕业设计题目是**「幻画空间」—— 一个面向二次元创作者的插画社区与约稿交易平台**。

简单来说，我做的是一个"国内版的 Pixiv + Skeb"。Pixiv 是日本最大的插画社区，Skeb 是日本最大的约稿平台。但它们没有中国本地化的支付方式，也没有中文界面。我的项目就是把这两者结合起来，做成一个完整的、适合国内用户使用的二次元创作交易平台。

项目体量方面：后端 **7 个微服务**，前端 **2 个单页应用**（用户端 + 管理端），总代码量约 **3 万行**。

## 二、项目定位与核心价值（1 分钟）

在开始讲技术之前，我先说明这个平台解决什么问题。

**用户侧**：一个二次元爱好者想找人画一张头像或插画，他需要找到合适的画师、沟通需求、支付费用、收到作品。目前国内没有专门的平台做这件事，大家只能在 QQ 群、贴吧、闲鱼上找，交易没有保障。

**画师侧**：画师需要一个展示作品、接单、管理约稿、收取报酬的地方。

**平台侧**：我作为平台方，需要撮合交易、保障双方利益、提供 AI 辅助工具提高效率。

所以我设计了三大核心体系：**作品社区**、**约稿交易**、**AI 辅助**。

## 三、技术架构总览（2 分钟）

（建议此时展示架构图）

技术选型上，后端我选择了 **Spring Cloud 微服务架构**，前端用 **Vue 3**。

整体架构分三层：

**最外层是网关层**：Spring Cloud Gateway 作为统一入口，所有请求先经过网关。网关做了三件事：JWT 身份认证、IP 限流、路由转发。

**中间是业务服务层**：6 个独立微服务，分别是用户服务、作品服务、约稿服务、通知服务、文件服务和 AI 服务。每个服务有自己的 MySQL 数据库，数据库之间不共享，通过 OpenFeign 做服务间调用，通过 RabbitMQ 做异步消息。

**最下面是基础设施层**：Nacos 做服务注册发现和配置管理，Redis 做缓存和限流计数，RabbitMQ 做消息队列，阿里云 OSS 做文件存储。

前端方面，我写了两套单页应用：用户端面向普通用户和画师，管理端面向平台管理员。都用了 Vue 3 + Pinia + Vue Router + Axios + Element Plus 这套技术栈。

## 四、核心功能演示（5 分钟）

下面我按用户视角，走一遍完整的核心流程。

### 4.1 作品社区 + AI 智能打标（重点）

用户打开网站，可以按标签、关键词搜索插画作品，按热度、时间排序，查看作品详情、点赞、收藏、评论。

**AI 智能打标是我这个项目最大的技术亮点**，我重点讲一下。

当一个画师上传一张插画时，后台会自动识别图片内容并生成中文标签。整个流程是这样的：

```
用户上传图片 → file-service 保存到阿里云 OSS → 返回图片 URL
→ artwork-service 保存作品记录 → 发送 RabbitMQ 消息
→ TaggingConsumer 消费消息 → HTTP 调用 ai-service (Python FastAPI)
→ DeepDanbooru 模型推理 → 返回标签列表 → 写入 artwork_tags 表
```

这里我做了几个关键设计：

**第一，模型选型**。我选用的是 DeepDanbooru，它是一个专门为二次元图片训练的深度学习模型，基于 ResNet 变体架构（ResNet-Custom-V3），输入 512×512 的 RGB 图像，输出 6000 多个标签的概率分布。它与 ImageNet 这类通用模型最大的不同在于：它的标签体系完全针对二次元场景——包含人物数量（1girl/2girls/solo）、发型（twintails/ponytail/ahoge）、服装（school_uniform/kimono/armor）、姿势（sitting/standing/running）、背景（sky/cityscape/classroom）、表情（smile/blush/tears）等等，覆盖了二次元插画的全部视觉特征维度。

**第二，推理优化**。原始模型输出的 6000 维概率向量包含了大量低置信度噪音。我做了三层过滤：首先，设置置信度阈值 0.4，低于这个值的标签直接丢弃；其次，过滤掉内容分级标签（如 rating:safe、rating:questionable），这些对用户没有意义；最后，过滤掉纯特殊字符的垃圾标签。经过三层过滤后，取置信度最高的 8 个标签返回，既保证了准确性，又避免了标签过多导致的信息过载。

**第三，中文翻译层**。DeepDanbooru 的原生输出是英文标签（如 "1girl"、"school_uniform"、"ahoge"），国内用户看不懂。我手写了一个 400+ 条目的内建翻译字典，覆盖了最常用的标签，包括人物数量、发型、服装、表情、姿势、背景、画风等分类。对于字典未覆盖的标签，还有一个基于规则的翻译回退机制。翻译结果会被缓存到内存中，避免重复计算。

**第四，异步解耦**。整个打标过程通过 RabbitMQ 异步触发。为什么这么做？因为一张 512×512 的图做完整的模型推理大约需要 0.5-1 秒，如果同步处理，用户上传后要等好几秒才能看到结果，体验很差。我通过消息队列把上传流程和打标流程解耦——用户上传后立即返回成功，打标在后台悄悄完成，前端周期性拉取或者通过 WebSocket 收到通知后刷新标签。

**另外，我还支持了批量打标**。用户一次最多提交 50 张图片，系统返回一个 job_id，用独立线程异步处理每张图片，处理进度可通过状态查询接口实时获取。

---

### 4.2 以图搜图

以图搜图是和 AI 打标配套的另一个智能功能。用户上传一张图片，系统找到作品库中视觉上最相似的画作。

技术实现上，我拆成了两个步骤：

**第一步：特征提取**。特征提取没有复用 DeepDanbooru，而是用了 **MobileNetV2**（在 ImageNet 上预训练），原因有两个：一是 MobileNetV2 提取的是通用视觉特征（构图、色彩、纹理），比 DeepDanbooru 的标签概率更适合做相似度匹配；二是 MobileNetV2 只有约 3.5M 参数，推理速度极快，单张图片不到 100ms。

具体流程是：图片预处理为 224×224 → MobileNetV2 提取 1280 维特征向量 → **分桶最大池化**降维到 256 维 → **L2 归一化**。这里的分桶最大池化是我自己设计的一个降维策略：把 1280 维按每 5 个一组分成 256 个桶，每个桶取最大值。这样做的好处是保留了最显著的特征响应，同时把维度降到适合存储在 MySQL 中的大小。

**第二步：相似度检索**。用户上传图片后，同样走一遍特征提取得到 256 维归一化向量。然后拿这个向量和数据库中已有作品的特征向量做 **余弦相似度** 计算。因为两个向量都已经 L2 归一化，余弦相似度就等于向量的内积，计算非常简单高效。按相似度降序排列，取 Top-K 返回。

整个流程支持两种输入方式：图片 URL 和直接上传文件，最大支持 10MB 的 JPG/PNG/GIF/WEBP 格式。

### 4.3 约稿交易（重点）

这是整个平台最核心的业务功能，我用一个完整的例子来说明：

**第一步**：用户看到喜欢的画师，点击"发起约稿"，填写需求描述、预算、参考链接，提交请求。此时约稿状态为"待报价"。

**第二步**：画师收到实时通知，查看需求后给出报价——包括总价、定金比例（默认 30%）、预计完成时间、允许修改次数。

**第三步**：用户看到报价，点击"接受并支付定金"，跳转到支付宝沙箱支付页面。支付成功后，约稿状态变为"已付定金"，资金冻结在画师钱包里暂不可提现。

**第四步**：画师开始创作，完成后交付作品链接，状态变为"已交付"。

**第五步**：用户确认收货，支付尾款。此时定金解冻、尾款入账，画师可以提现。约稿完成。

这个流程的关键设计在于**资金安全**：定金在交付完成前是冻结的，画师不能拿钱跑路。如果画师取消约稿，定金和尾款全退。如果用户取消，定金不退——这是行业惯例，保护画师的时间成本。

### 4.4 支付体系

支付方面我做了几个比较细致的设计：

- **平台手续费 5%**，VIP 减免 5%，SVIP 减免 10%——这是商业变现的基础
- **优惠券只能用于尾款**，不能用于定金——防止用户用券付定金然后取消套现
- **30 分钟订单超时**，超过 30 分钟未支付的订单自动关闭
- **已有待支付订单时直接复用**，不会重复创建

### 4.5 AI 智能客服

平台内置了一个 AI 客服助手"小幻"，用户可以在右下角随时唤起对话。我来详细讲一下技术实现：

**模型对接**：后端对接的是火山引擎 ARK 平台的 LLM API（OpenAI 兼容协议），通过官方 OpenAI Python SDK 调用。我单独写了一个 httpx 客户端处理连接，绕过了系统代理可能导致的请求失败问题。

**System Prompt 工程**：这是 AI 客服最核心的部分。我手写了一套 600+ 字的平台知识库 Prompt 注入到 system 消息中，内容包括：所有功能模块的描述（作品、约稿、会员、比赛等）、每个页面的路由路径、回复规则和语气风格。这样当用户问"约稿怎么付款"时，LLM 能基于真实的平台功能来回答，而不是瞎编。举个例子，Prompt 里明确写了"约稿流程：提交请求 → 画师报价 → 用户接受并支付 → 画师创作 → 交付作品 → 用户确认"，所以 AI 给出的描述和平台实际功能是一一对应的。

**流式输出**：为了提高体验，我实现了 SSE（Server-Sent Events）流式接口。LLM 以 stream=True 模式调用，每生成一个 token 就立即推送给前端，前端拼接成打字机效果。用户不需要等整个回答生成完才能看到第一个字。

**智能追问推荐**：每次 AI 回复后，系统会根据对话内容中的关键词（约稿/支付/画师/作品等）自动生成 3 个推荐追问，帮助用户更快地找到需要的信息。比如用户问了约稿相关的问题，推荐追问可能就是"约稿流程是怎样的？""约稿如何付款？"

**会话管理**：支持多轮对话，保留最近 10 轮历史上下文。会话记录存储在数据库中，用户可以回溯之前的咨询记录。

## 五、技术亮点深入（3 分钟）

下面我挑 4 个最有技术含量的点展开讲。

### 5.1 AI 智能打标：从模型推理到工程落地

这是我认为整个项目最有技术深度的部分。我不仅仅是调了一个现成的模型 API，而是从模型加载、推理管线、标签过滤、中文翻译到异步架构做了一套完整的工程方案。

**模型层面**：DeepDanbooru 是一个基于 ResNet-Custom-V3 的多标签分类模型。它和普通分类模型的最大区别是——它不是"这张图是猫还是狗"的单选，而是"这张图里同时有 1 个女孩 + 长发 + 校服 + 微笑 + 樱花背景"的多标签预测。输出层用的是 sigmoid 而非 softmax，每个标签的预测是独立的概率值。这意味着 6000 多个标签之间不是互斥关系，一张图可以同时命中有多少个标签。

我在加载模型时做了兼容设计——代码自动检测模型目录下的文件，支持 model-resnet_custom_v3.h5、v4 等多个版本，加载时不编译（compile=False），因为只做推理不需要反向传播图，这样可以节省约 30% 的加载时间和内存占用。

**推理管线**：图片进入模型前要经过严格的预处理——强制转 RGB（防止 PNG 的 RGBA 四通道导致维度错误）、LANCZOS 算法缩放到 512×512、归一化到 [0,1]、增加 batch 维度。这些步骤看起来简单，但任何一步出错都会导致模型崩溃或输出异常。

**标签过滤是我自己设计的**：原始模型输出 6000 多个标签全部带概率值，但其中有大量噪音。我设计了三级过滤策略：
1. 置信度阈值 0.4——低于此值的丢弃（经验值，经过多轮测试确定的平衡点）
2. 语义过滤——自动跳过内容分级标签（rating: 开头）和纯符号垃圾标签
3. Top-K 截断——只保留置信度最高的 8 个标签，确保 UI 上不信息过载

**翻译层**：DeepDanbooru 原生的英文标签对国内用户不友好。我手写了一个内建翻译字典，覆盖了 400+ 最常用标签。这个字典我分了几十个类别来组织：人物数量（solo→单人）、发型（twintails→双马尾，ahoge→呆毛）、服装（school_uniform→校服，kimono→和服）、表情（blush→脸红，teary_eyes→泪眼）、姿势、背景、画风等等。字典覆盖不到的标签用规则回退（如去掉下划线、首字母大写等）。

**工程层面**：整个打标链路通过 RabbitMQ 异步触发。为什么这是必要的？因为深度学习模型推理是 CPU/GPU 密集型操作，如果同步处理会严重拖慢用户上传体验。异步解耦后，用户上传作品 → 立即返回成功 → 打标在后端悄悄完成 → 写入数据库 → 前端下次刷新就能看到标签。另外，批量打标接口支持最多 50 张图片并发提交，用 Python threading 异步处理，通过 job_id 查询进度。

### 5.2 以图搜图：双模型协同的视觉检索

以图搜图这个功能，我选择了一条和 AI 打标不同的技术路线。

**为什么不直接复用 DeepDanbooru？** 因为 DeepDanbooru 提取的是语义标签概率（"这张图有 95% 概率是长发"），而搜图需要的是视觉相似度（"这两张图的整体构图和色彩像不像"）。标签概率向量用来做相似度匹配效果很差——两张完全不同风格的图可能都包含"1girl"和"long_hair"，它们就会被认为是相似的。

**所以我的方案是双重特征提取**：
- DeepDanbooru 负责语义理解（这张图画了什么内容）
- MobileNetV2 负责视觉匹配（这张图的视觉风格是什么）

MobileNetV2 是在 ImageNet 百万级图片上预训练的，它见过的图片类型足够多，能自然地学会区分二次元插画、真实照片、图表截图等不同视觉类型。它的 1280 维 GlobalAveragePooling 输出本质上是一个压缩的视觉"指纹"。

**降维策略**：1280 维向量太大了，不适合存储在 MySQL 中做实时检索。我设计了一个**分桶最大池化**的降维方法：把 1280 维按顺序分成 256 个桶（每桶 5 个值），每个桶取最大值。这样做比直接取前 256 个值或平均池化效果更好，因为它保留了每个局部区域的显著特征。最后做 L2 归一化，使得向量模长为 1，余弦相似度计算退化为向量内积——O(n) 的复杂度非常高效。

**检索流程**：用户上传图片 → MobileNetV2 提取 256 维归一化向量 → 同时返回 DeepDanbooru 标签结果 → 拿特征向量和数据库中所有作品的向量做内积 → 按相似度降序返回。每张作品在首次发布时就提取好特征向量存储，检索时不需要重新推理。

### 5.3 约稿状态机 + 退款规则

约稿服务是整个系统业务逻辑最复杂的部分。我设计了一个严格的**状态机**：

```
PENDING（待报价）→ QUOTED（已报价）→ DEPOSIT_PAID（已付定金）
→ IN_PROGRESS（创作中）→ DELIVERED（已交付）→ COMPLETED（已完成）
```

每个状态变更操作都有**双重校验**：
- 当前用户是否有权操作（是委托方还是画师）
- 当前状态是否允许此操作

**退款规则**是最复杂的业务逻辑：

```
取消方       定金处理     尾款处理
─────────────────────────────────
画师取消     全额退还     全额退还   （画师违约）
委托方取消   不予退还     退还尾款   （委托方违约）
管理员取消   可灵活决定   可灵活决定  （纠纷处理）
```

这个规则参照了行业惯例——画师违约全退、委托方违约定金不退。技术实现上，取消操作的 refundPayment 方法会自动遍历所有已支付订单，根据 refundDeposit 参数决定是否跳过定金退款，逐笔调用支付宝退款 API。同时，退定金时还会调用户服务的 cancelWalletFreeze 接口解冻画师钱包。

### 5.4 Token 无感刷新 + 请求队列

前端 Axios 拦截器里实现了一个经典的请求队列机制：

```
第一个 401 → 发 POST /auth/refresh 拿新 token
其他 401 → 把请求放进 failedQueue 排队等待
刷新成功 → 用新 token 逐一重放队列中的请求
刷新失败 → 清空队列，跳转登录页
```

这个设计避免了一个常见问题：如果页面上同时有 5 个请求返回 401，并发刷新会产生竞态和资源浪费。用了队列后，只有第一个触发刷新，其余 4 个等待结果。队列用 Promise 的 resolve/reject 控制——每个排队的请求返回一个新的 Promise，在刷新完成后批量 resolve。

## 六、技术难点与解决方案（2 分钟）

### 难点 1：AI 模型的跨语言服务调用

DeepDanbooru 是基于 TensorFlow/Python 生态的模型，而我的后端是 Java Spring Cloud。如果在 JVM 里直接跑 TensorFlow 模型（比如用 Deep Java Library），会导致启动极慢、内存占用高、而且难以利用 GPU 加速。我的方案是**独立部署 Python AI 服务**，用 FastAPI 提供 RESTful 接口，Java 后端通过 HTTP 调用。这个方案的好处很明显：Python 服务可以独立扩缩容、独立更新模型、不拖累 Java 服务启动。但代价是多了一跳网络延迟——经过测试，在同机部署的情况下降幅很小，模型推理本身才是瓶颈。

### 难点 2：标签翻译的覆盖率问题

DeepDanbooru 有 6000+ 标签，但很多是低频或高度特化的（比如某个具体动漫角色的名字），不可能全部手动翻译。我采取了**分层翻译策略**：第一层是 400+ 高频标签的内建字典（手写翻译质量最高），第二层是规则翻译（去下划线、首字母大写），第三层回退显示原文。最终，前两层覆盖了实际使用场景中约 85% 的标签，剩下 15% 的罕见标签直接显示英文，对用户体验影响很小。

### 难点 3：支付回调的幂等性

支付宝异步通知可能因为网络原因重复发送。我的处理方式是：收到通知后先检查订单状态，如果是 PAID 直接返回 success，不做任何业务操作。会员支付场景下，先执行升级逻辑再标记 PAID——如果升级失败返回 failure 给支付宝，支付宝会按策略重试。这保证了"升级"和"标记已支付"的原子性。

### 难点 4：跨服务的数据一致性

支付成功 → 更新约稿状态 → 冻结画师钱包，这涉及约稿服务和用户服务的联动。我采用**最终一致性**的策略：支付宝回调中按顺序执行这些操作，如果关键步骤失败，返回 failure 让支付宝重试。对于非关键操作（如发通知、更新搜索索引），用 try-catch 包裹，失败只记日志不影响主流程。没有引入 Seata 这类分布式事务框架，在当前的业务规模下，这种"尽力交付 + 失败重试"的模式是够用的。

## 附：预判答辩提问 & 建议回答

### Q1：为什么选 DeepDanbooru 而不是其他模型？你做了什么改进？
> DeepDanbooru 是目前唯一开源的大规模二次元图片多标签分类模型，标签体系覆盖了 6000+ 个维度，比 Waifu Diffusion 的标签体系更全面。我在其基础上做的改进包括：第一，实现了从预处理到推理到过滤到翻译的完整工程管线，让模型能在生产环境中稳定运行；第二，设计了三级标签过滤策略（置信度阈值 + 语义过滤 + Top-K 截断），把 6000 维的噪声输出提炼为 8 个高质量标签；第三，手写了 400+ 条目的中文翻译字典，解决了英文标签对国内用户不友好的问题。

### Q2：以图搜图为什么不直接用 DeepDanbooru 做特征提取？
> 因为两种任务的目标不同。DeepDanbooru 输出的是语义标签概率分布——"这张图里有什么内容"。但搜图需要的是视觉相似度——"哪张图看起来像这张"。两张风格迥异的图可能有相似的标签（都有"1girl + long_hair"），但视觉上完全不同。我选 MobileNetV2 是因为它在 ImageNet 上预训练过，学到的卷积特征对色彩、构图、纹理更敏感，更适合做视觉相似度匹配。而且我设计了一个分桶最大池化降维策略，把 1280 维压缩到 256 维，既保留了显著特征，又方便存储和计算。

### Q3：为什么用 4 个数据库而不是 1 个？
> 这是微服务 DB-per-service 的最佳实践。如果所有服务共享一个数据库，微服务的独立部署和扩展就没有意义了——你改了用户表的字段，约稿服务也要跟着改，这就又回到了单体架构的耦合问题。每个服务有自己的数据库，服务之间只能通过 API 通信，真正做到了数据边界的隔离。

### Q4：AI 客服如果用通用大模型，怎么保证回答的准确性？
> 核心靠 System Prompt 工程。我手写了一套 600+ 字的平台知识库，详细描述了每个功能的流程、路由地址、规则限制，直接注入到 LLM 的 system 消息里。这样 LLM 在生成回答时会优先基于我提供的知识库，而不是它的训练数据。另外，回复规则里明确要求"不编造不存在的功能"，遇到无法解答的问题会引导用户联系人工客服。这本质上是一个 RAG（检索增强生成）的简化版——知识库就是我的检索文档。

### Q5：分布式事务怎么处理的？
> 目前没有用 Seata 之类的分布式事务框架，用的是最终一致性的思路。比如支付成功后更新约稿状态和冻结画师钱包，是顺序执行的，如果钱包操作失败，返回 failure 让支付宝回调重试。发送通知这类非关键操作，失败了只记日志不影响主流程。

### Q6：JWT 和 Session 相比有什么优劣？
> JWT 的优势是无状态，网关验证签名就行，不需要查 Redis 或数据库。缺点是签发后无法主动失效——比如管理员封禁用户，已经签发的 token 还是有效的。我的补偿方案是 token 有效期设得比较短，配合 refresh token 做到期刷新。更完善的方案可以引入 token 黑名单。

### Q7：你的系统能支持多少并发？
> 网关层没有做极限压测，但架构上是支持水平扩展的。每个微服务都可以部署多个实例，Nacos 做负载均衡。网关层有 Redis 限流保护，高并发场景下可以把热点数据（如作品列表）放到 Redis 缓存里。如果要上生产，还需要加上 Sentinel 做熔断降级。AI 服务目前是单实例，但也可以部署多个实例 + Nginx 负载均衡。

### Q8：移动端适配做了吗？
> 前端做了响应式布局，768px 断点以下会自动切换为单列布局。但目前主要还是桌面端体验，移动端没有单独开发。如果要上移动端，可以考虑把 Vue 项目用 Capacitor 打包成 APP，或者用 uni-app 重写。

### Q9：优惠券为什么不能用于定金？
> 这是一个防止套利的业务设计。假设优惠券可以用在定金上，用户可以领券 → 付定金 → 立刻取消约稿 → 因为是委托方取消，定金不退 → 平台退还了优惠券抵扣的金额 → 优惠券就变成了可提现的现金。限制优惠券只能用于尾款，避免了这个问题。

### Q10：数据库表结构能简单说一下吗？
> 每个服务有自己的核心表。比如约稿服务有 commissions（约稿主表）、payment_orders（支付订单）、coupons（优惠券）、user_coupons（用户领券记录）、commission_messages（约稿内消息）。作品服务有 artworks（作品）、tags（标签）、artwork_tags（作品标签关联）、artwork_features（特征向量存储）、comments（评论）、likes（点赞）、favorites（收藏）。这些都是按照业务域来划分的，符合 DDD 的思想。
